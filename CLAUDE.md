# CLAUDE.md

このファイルは、このリポジトリ（`share-re/manage-task`）で作業する際に Claude Code が
参照し、**厳守**する指示書です。

## プロジェクト概要
進捗管理アプリ。ログインして利用し、ログイン後のトップ画面から各機能
（進捗管理・AI内田さん・植林）へ移動できるようにする。
今回はログイン＋トップ画面（各機能への導線まで）を対象とし、各機能の中身は未実装。

## 開発工程（この順で進める）
各機能・各 Issue は、次の 4 工程を **この順番** で進める。
**前の工程が終わり、認識が合ってから次へ進む**（工程を飛ばさない。特に「設計」を飛ばさない）。

1. **企画検討・要件定義** … 「何を作るか／なぜ作るか」を決める。
   Issue 化し、必要なら `docs/` に要件をまとめる。「完成したら何がどうなっていれば OK か
   （期待する結果）」まで言語化する。
2. **設計** … 「どう作るか」を決める。データ構造（DB のテーブル・列）、画面 / UI、
   処理の流れ、影響範囲を実装前に固め、認識を合わせる。
3. **開発** … 設計に沿ってコードを書く（詳細は「## 開発の流れ」）。
   作業ブランチで実装し、こまめにコミットして PR を出す。
4. **公開** … レビュー OK の PR を `main` にマージし、本番サイトへ反映する
   （詳細は「## 公開（Vercel）」）。

## 認証
- ユーザー認証には **Supabase の認証機能** を使う。
- 認証方式の詳細はユーザーと相談しながら決める。

## GitHub 運用ルール（厳守）
- このアプリの運用は GitHub を利用する。
- **`main` は常に動く完成版**。`main` へは直接コミット・直接 push しない。
- 作業は担当ごと・機能ごとに分ける。作業ブランチは必ず `main` から切り、その中で開発する。
- **`main` への反映は必ず PR（プルリクエスト）経由**とする。

### ブランチ命名
- ログイン＋トップ画面: `feature/login`
- 以降、他の機能もそれぞれ `feature/<機能名>` の形で切る。

## 開発の流れ
1. `main` を最新にする
2. `main` から作業ブランチを切る
3. そのブランチで開発し、こまめにコミットする
4. 区切りがついたら PR を出す
5. ほかのメンバーが軽くレビューする
6. OK なら `main` にマージ → Vercel が自動で本番へ反映

## 公開（Vercel）※はじめての人向けの解説

「公開」とは、書いたコードを **実際にインターネット上で誰でも使えるサイトに反映すること**。
このプロジェクトでは **Vercel** というサービスが、その作業を **自動で** やってくれる
（自分でサーバーにファイルをアップロードする必要はない）。

### まず用語を 3 つだけ
- **本番サイト** … `main` ブランチの内容がそのまま公開されている“本物”のサイト。
  ここは常に動く完成版でなければならない（だから `main` に直接コミットしない）。
- **デプロイ** … コードを「動く形」に変換して、サイトへ反映する作業のこと。Vercel が自動でやる。
- **プレビュー URL** … PR ごとに Vercel が自動で作る“お試しサイト”の URL。
  マージ前に実際の画面を確認でき、本番には影響しない。

### 公開までの流れ（ステップ）
1. 作業ブランチで開発し、**PR を出す**。
2. Vercel が自動で **プレビュー URL** を作る → その URL で画面を確認し、レビューを受ける。
3. レビュー OK → PR を `main` に **マージ**する（GitHub 上のマージボタンを押すだけ）。
4. マージをきっかけに、Vercel が **自動でデプロイ**（ビルド＋本番反映）。数分で本番サイトに出る。
5. **本番サイトを開いて**、意図どおり動くか最終確認する。

### もし本番で問題が出たら
慌てない。問題を起こした PR を **元に戻す（revert）PR** を作ってマージすれば、
本番を以前の状態へ戻せる。まず戻して安全にしてから、落ち着いて原因を直す。

## コーディング / コミュニケーション規約
- 説明・レビューコメント・チャット応答は **日本語** で書く。
- コード内コメント・識別子（変数 / 関数 / クラス名）・コミットメッセージは **英語** で書く。
- 結論を先に述べ、根拠は簡潔に添える。
- 不明点は推測で埋めず、先に確認する。

## ポリシー（厳守）
- `main` への直接 push は禁止（必ず PR 経由）。
- ファイルの削除は必ず事前確認を行う（許可なしの削除は禁止）。
- パスワードファイル（`.env` 等）の読み取り禁止。
- コードやファイルの外部アップロード禁止。
- この `CLAUDE.md` の改ざん・上書きは、ユーザーの許可なく行わない。

## 📖 くわしい内容

`docs/requirements-ai-uchida-v2.md` の **Q3** に対応する Issue。
**前提: #77（RAG）が完了していること**（社内資料という土台が無いと成立しない）。

RAG で社内資料を検索できるようになったら、その使い方として
「一般論だけでなく、内田さん／この会社ならではの見方まで相談できる」体験を作る。

- 社内資料が根拠になり、かつ**一般論と社内／内田さんの見方が食い違うとき**は、その違いが分かるように答える:

  ```
  一般的にはこう言われがち → でも、うちの場合は（社内資料に基づく見方）
  ```

- 見た目を厳密に2段に割ることは求めない（内田さんの口調のまま自然に織り込む）。
  一般論と社内の見方がほぼ同じで対比の意味が薄いときは、無理に分けない。
  ※当初は「常に見出しで2段」を条件にしていたが、人格プロンプトと衝突し安定しなかったため緩めた（下の「補足・メモ」参照）
- 該当資料が無いときは**従来どおり応答する**（#77 の検索と同じ扱い。
  雑談か業務相談かの区別判定はしない）
- 社内の見方として述べる**事実は資料に忠実**とする。言い回し・たとえ方を内田さんらしくするのは許容するが、
  資料に無い体験談を創作してはならない
- 本番操作・セキュリティ・お金・評価や人間関係などの**重い判断**は、資料の有無にかかわらず
  断定せず、本人・担当者への確認を勧める（v1.0 PS-07 の継続）

実装は主に人格プロンプト（`src/lib/uchidaPrompt.ts`）と、RAG 検索結果の
プロンプトへの渡し方の調整になる見込み。

## 📌 補足・メモ

- 「知ったかぶりをしない」「作り話をしない」は v1.0 PS-06 と直近のプロンプト改善方針の継続。
  本 Issue はそれを「一般論と社内の見方の違いを示す」形に整理するもの
- 判定用の試験質問は `docs/uchida-interview-sheet.md` の想定問答を流用しつつ、
  「資料に該当がある質問／無い質問」を数問追加する（要件定義書 §10 冒頭）
- ブランチ名の例: `feature/assistant-two-tier-answer`

### 2026-07-22 受け入れ基準の変更（当初「常に見出しで2段」→「違いが伝わればよい」）
- 当初案は「社内資料がヒットしたら、回答を必ず見出しで2段（一般論／内田論）に割る」だった。
- プロンプトで指示を強めても、賢いモデル（gemini-2.5-flash）でも安定して2段にならなかった。
  原因は人格プロンプトの『結論から短く・繰り返さない・自然に会話』方針との衝突で、
  モデルが一般論の段を“冗長な前置き”として削り、いきなり社内の結論に飛ぶため。
- 実際の回答は資料に忠実で品質も十分だったので、"見た目の2段"の強制はやめ、
  「一般論と社内の見方が食い違うときに、その違いが分かるように答える（自然に織り込む形でよい）」に緩めた。
- 要件定義書 `docs/requirements-ai-uchida-v2.md` の FR-Q3-01・§10 も同様に更新済み。

DAY 1 ─ Chapter 01 ⏱ 1.5h Webの仕組みを理解する ブラウザとサーバーがどう会話しているのか

### 1-1. インターネットとWebの違い

「インターネット」と「Web」は混同されがちですが、別のものです。**インターネット**は世界中のコンピュータを繋ぐ「道路網」のようなもの。**Web**はその道路網の上で動く「配送サービス」の一つです。メールやファイル転送も同じ道路網（インターネット）を使いますが、Webとは別の仕組みです。

［図（テキスト抽出）：インターネット（道路網） / 🌐 Web (HTTP) / Webページの閲覧 / 📧 メール (SMTP) / メールの送受信 / 📁 FTP / ファイルの転送 / これらはすべて、インターネットという共通の基盤の上で動いています / この研修では「Web（HTTP）」を扱います ☝］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

### 1-2. クライアントとサーバー

Webの世界には、大きく2つの登場人物がいます。

- **クライアント（ブラウザ）** ─ 「このページを見せて！」とお願いする側
- **サーバー** ─ 「はい、どうぞ」とページを返す側

この「お願い」と「返事」のやり取りをHTTPプロトコルと呼びます。Webのすべての通信はこの仕組みで成り立っています。

DAY 1 ─ Chapter 02 ⏱ 2.0h HTMLの基礎 Webページの「骨組み」を作る言語

### 2-1. HTMLとは何か

**HTML（HyperText Markup Language）**は、Webページの構造を記述するための言語です。プログラミング言語とは異なり、「ここは見出し」「ここは段落」「ここは画像」というように、コンテンツに**意味（役割）**を付ける「マークアップ言語」です。

> **📌 たとえ話**
>
> HTMLは建物でいう「設計図・骨組み」です。部屋の配置（構造）は決まりますが、壁の色や家具の配置（見た目）はCSSが担当します。

### 2-2. タグの基本

HTMLは**タグ**でコンテンツを囲むことで構造を表現します。

DAY 1 ─ Chapter 04 ⏱ 1.5h フォームと入力 ユーザーからデータを受け取る仕組み

### 4-1. フォームは「封筒」─ データをまとめてサーバーに届ける

Webページでユーザーがデータを入力・送信するには**フォーム**を使います。ログイン画面、検索ボックス、お問い合わせフォームはすべてこの仕組みです。

フォームを理解するには、**「封筒」**にたとえるのが一番わかりやすいです。

［図（テキスト抽出）：フォーム ＝ 封筒 / 宛先（action） / greet.jsp / ← どのURLに届けるか / 配送方法（method） / POST / ← GET or POST / 📋 中身（name=value カード） / username=田中 / age=25 / 送信 / HTTP / リクエスト / 🗄️ サーバー / 受取口：greet.jsp / POST で受け付け中 ✓ / 受取口：search.jsp / POST で受け付け中 / 受取口：hello（Servlet）］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 4-1：フォームは「封筒」─ 宛先・配送方法・中身のカードを持って、サーバーの受取口に届けられる

> **📌 封筒の3要素**
>
> **① 宛先（action）** ─ どのURLに届けるか  
> **② 配送方法（method）** ─ GET（URLの表面に書く）か POST（封筒の中に入れる）か  
> **③ 中身（name=value）** ─ 各入力欄が生み出す「名前=値」のペア  
> 送信ボタンを押すと、この封筒が**HTTPリクエスト**としてサーバーのURLに届けられます。

### 4-2. name=value のしくみ ─ 封筒の中身

フォームの各入力欄は、`name` 属性と入力された値で **name=value** というペア（カード）を1枚ずつ生み出します。送信時にこれらが `&` で連結されてサーバーに届きます。

［図（テキスト抽出）：ユーザーの入力 → name=value ペアの生成 / 📝 フォーム（ユーザーが入力） / 名前： / 田中 / 年齢： / 25 / 部署： / 開発部 ▼ / 送信 / 変換 / 📋 生成される name=value カード / name= "username" value= "田中" / name= "age" value= "25" / name= "dept" value= "dev" / ↓ & で連結されてサーバーに届く / username=田中&age=25&dept=dev］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 4-2：入力 → name=value ペア → & で連結されてリクエストとして送信される

#### 📮 POST＝封筒、GET＝ハガキ

データの送り方には `POST` と `GET` の2種類があります。この2つの違いは、**郵便物**にたとえるとイメージしやすいです。

［図（テキスト抽出）：POST は封筒、GET はハガキ / 📮 POST ＝ 封筒 / 宛先: /greet.jsp / 中身は封をして見えない 🔒 / username=田中&age=25 / → URLには表示されない / 📬 GET ＝ ハガキ / 宛先: /greet.jsp / ?username=田中 / &age=25 / &dept=dev / 表面に全部書かれて丸見え 👀 / → URLに付いて表示される / ブラウザの履歴にも残る / 書ける文字数にも限界あり］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 4-3：POST は封筒（中身が隠れる）、GET はハガキ（中身が表面に見える）

#### 🔍 GET はブラウザのアドレスバーに見える

`method="get"` で送信すると、データは URL の末尾に `?` に続けて付きます。

**GET で送信したときのURL例**

    http://localhost:8080/training/greet.jsp?username=田中&age=25&dept=dev
                                             ↑ ここから全部、ブラウザに見えてしまう

ブラウザのアドレスバーで実際に見えるので、学習中の動作確認には便利です。ただし、見えるということは**他人にも見える**ということです。

#### 📏 GET には文字数の制限がある

ハガキに書ける文字数に物理的な限界があるように、GET で送れるデータ量にも制限があります。

［図（テキスト抽出）：GET の URL 長さ制限（目安） / ~2,000 文字 / 安全ゾーン / どのブラウザでも確実に動く / ~8,000 文字 / 注意ゾーン / 一部環境で切れる可能性 / 8,000 文字以上 / 危険ゾーン / 多くの環境で 414 エラー / ※ 制限はブラウザ・サーバーの実装に依存します。 / 一般に 2,000 文字以内 に収めるのが安全です。］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 4-4：GET の URL 長さは環境依存 ─ 2,000文字を超えると不安定に

一方、`POST` は封筒の中（HTTPリクエストのボディ）にデータを入れるので、**文字数の制限は実質ありません**（サーバー側の設定次第で、数MB〜数十MBまで可能）。

#### 📊 GET と POST の使い分け

|                        | GET（ハガキ）                   | POST（封筒）                           |
|------------------------|---------------------------------|----------------------------------------|
| データの見え方         | URLに表示される（丸見え）       | URLには表示されない（見えない）        |
| ブラウザの履歴         | データごと残る                  | データは残らない                       |
| 文字数制限             | 2,000文字が安全圏               | 実質制限なし                           |
| ブックマーク           | できる（URL含めて保存）         | できない                               |
| 向いている用途         | 検索、ページ指定、共有したいURL | 登録、ログイン、長文投稿、ファイル送信 |
| パスワード等の秘密情報 | ❌ 絶対使わない                 | ✓ こちらを使う                         |

> **📌 迷ったら POST**
>
> ユーザーからの入力を受け取る場合、基本は `POST` を使いましょう。`GET` は「検索結果をURLで共有したい」「ブックマーク可能にしたい」など、**URLにデータが含まれて嬉しい**場面で選びます。

### 4-3. サーバー側の「受取口」─ URLとメソッドで待ち構える

封筒を届ける相手（サーバー）側には、**特定のURLで、特定のメソッド（GET/POST）のリクエストを待ち構える「受取口」**があります。

［図（テキスト抽出）：サーバーの「受取口」─ URL ＋ メソッドのペア / 🗄️ サーバー（Tomcat） / 受取口 ① / URL: /training/greet.jsp / GET/POST / 受取口 ② / URL: /training/calc.jsp / POST / 受取口 ③（DAY 4 Servlet） / URL: /training/search / POST / 受取口 ④（DAY 4 Servlet） / URL: /training/hello / GET］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 4-5：サーバーには「このURLに、このメソッドで来たら処理する」という受取口が並んでいる

DAY 2のJSPではファイル自体がURLとなり、GET/POSTの両方を暗黙的に受け付けます。DAY 4のServletでは `doGet()` / `doPost()` メソッドで**明示的に**どちらで受けるかを宣言します。

### 4-4. 整合が取れて初めて動く ─ 「約束ごと」の考え方

ここが最も重要なポイントです。フォーム（送る側）とサーバーの受取口（受け取る側）は、**3つの項目が一致**して初めて正しく動きます。1つでもずれるとエラーになります。

［図（テキスト抽出）：送る側と受け取る側の「約束ごと」 / 📝 送る側（HTML フォーム） / action= "greet.jsp" / method= "post" / name= "username" / name= "age" / 🗄️ 受け取る側（サーバー） / URL: /greet.jsp / POST で受け付ける / getParameter( "username" ) / getParameter( "age" ) / ✓ / すべてが一致 → 正常に動作 🎉 / 1つでもずれる → エラー（404 / 405 / null）］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 4-6：送る側と受け取る側の「約束ごと」─ 3つの整合が取れて初めて機能する

| 一致すべき項目 | 送る側（form）       | 受け取る側（サーバー）       | ずれると…              |
|----------------|----------------------|------------------------------|------------------------|
| ① URL          | `action="greet.jsp"` | `/training/greet.jsp` が存在 | 404 Not Found          |
| ② メソッド     | `method="post"`      | POST で受け付ける設定        | 405 Method Not Allowed |
| ③ パラメータ名 | `name="username"`    | `getParameter("username")`   | null（値が取れない）   |

> **📌 これがWeb開発の基本設計**
>
> フォーム（HTML）とサーバープログラム（JSP/Servlet）は**セットで設計**するものです。どちらかだけ作っても動きません。「送信する側」と「受け取る側」で**約束ごと（整合）**を揃えて実装することで、はじめてWebの機能が成り立ちます。  
> DAY 2 の JSP では `request.getParameter()` で、DAY 4 の Servlet では `doGet()` / `doPost()` でこの受け取りを実装します。

### 4-5. フォームの基本構造

ここまでの概念を踏まえて、実際のコードを見てみましょう。封筒のたとえがそのままコードに対応しています。

**form の基本**

    <!-- 封筒の作成：宛先(action)と配送方法(method)を指定 -->
    <form action="greet.jsp" method="post">

      <!-- 中身のカード①：name="username" -->
      <label for="username">名前：</label>
      <input type="text" id="username" name="username">

      <!-- 中身のカード②：name="age" -->
      <label for="age">年齢：</label>
      <input type="number" id="age" name="age">

      <!-- 送信ボタン：封筒を届ける -->
      <button type="submit">送信</button>

    </form>

### 6-6. WEB-INF と META-INF の役割

フォルダ構成に出てくる `WEB-INF/` と `META-INF/` は、どちらも**ブラウザから直接アクセスできない特別なフォルダ**です。ここにはアプリの内部構造やセキュリティ上、外に見られたくないファイルを置きます。

#### 🛡️ WEB-INF ─ 外から見られたくないものを入れる場所

ブラウザで `http://localhost:8080/training/WEB-INF/web.xml` と打っても **404 エラー**になります。これは Servlet の仕様で定められている保護です。

| サブフォルダ       | 入れるもの                       | なぜ隠すのか                                  |
|--------------------|----------------------------------|-----------------------------------------------|
| `WEB-INF/web.xml`  | アプリの設定ファイル             | 中身が見えると攻撃のヒントになる              |
| `WEB-INF/classes/` | コンパイル済み Servlet（.class） | Java バイトコードを外に晒さない               |
| `WEB-INF/lib/`     | 外部ライブラリ（.jar）           | 同上                                          |
| `WEB-INF/jsp/`     | Servlet から forward 専用の JSP  | **ユーザーが直接URLで開けないようにするため** |

**📌 なぜ JSP を WEB-INF の中に入れるのか？**

DAY 4 で学ぶ **MVC パターン**では、「処理は Servlet、表示は JSP」と役割分担します。このとき、ユーザーが JSP に直接アクセスできてしまうと、**Servlet での処理をスキップされてしまう**ため、JSP を WEB-INF の中に置いて Servlet 経由でしか表示できないようにします。

```
// 正しい流れ
ユーザー → /training/search → SearchServlet（検索処理）
                                    ↓ forward
                            WEB-INF/jsp/searchResult.jsp で表示

// WEB-INF の保護がなければ…
ユーザー → /training/jsp/searchResult.jsp を直接アクセス
         ↓ Servletの処理をスキップしたので結果がない
         ↓ 画面が空っぽでエラー！
```

WEB-INF 内の JSP は **URL から直接開けない**ため、必ず Servlet 経由になることが保証されます。

#### 📦 META-INF ─ Tomcat への指示書

`META-INF/` も直接アクセスできないフォルダで、**Tomcat 固有の設定ファイル**を置く場所です。最も重要なのが `context.xml`：

**META-INF/context.xml ─ Tomcat への追加指示**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Context reloadable="true">
</Context>
```

`reloadable="true"` と設定すると、Tomcat が WEB-INF/classes を監視して **.class の更新を自動検知**してくれます。研修中に Servlet を書き換えるたびに Tomcat を再起動せずに済むので便利です（解答ZIPにはこの設定が入っています）。

**🔑 2つの特別なフォルダのまとめ**

|                  | WEB-INF                  | META-INF          |
|------------------|--------------------------|-------------------|
| 役割             | アプリの内部ファイル格納 | Tomcat 固有の設定 |
| 仕様上の位置づけ | Servlet 仕様で標準化     | Tomcat の独自拡張 |
| 代表ファイル     | web.xml, classes/, jsp/  | context.xml       |
| 直接アクセス     | ❌ 不可（404）           | ❌ 不可（404）    |

### 7-1. JSPの3つの記法

JSPでは、HTMLの中に特別なタグを使ってJavaのコードを埋め込みます。

| 記法                | 書き方             | 役割                     |
|---------------------|--------------------|--------------------------|
| スクリプトレット    | `<% Javaコード %>` | Javaの文を実行する       |

### 7-2. はじめてのJSP

**hello.jsp**

```jsp
  <!-- 式：変数の値を出力 -->
```

### 8-1. if文（条件分岐）

「もし〇〇なら△△を表示する」という処理には `if` 文を使います。

**if文の例**

    <%

> **📌 HTMLとJavaが交互に出てくる**
>

### 9-1. リクエストはどう扱われる？ ─ Webサーバーとアプリケーションサーバー

Ch.4 で「フォームは封筒」と学びました。その封筒（リクエスト）が実際にサーバーに届いたあと、どのように処理されるのかを見ていきましょう。

#### 🏢 サーバーは「二段構え」になっている

実は多くのWebシステムでは、リクエストは **2種類のサーバー** を経由して処理されます。

図 9-1：Webサーバー（受付） → アプリケーションサーバー（処理）の役割分担

**📌 2種類のサーバーの役割分担**

|              | Webサーバー（Apache等）                                             | アプリケーションサーバー（Tomcat等）                  |
|--------------|---------------------------------------------------------------------|-------------------------------------------------------|
| 得意なこと   | 静的ファイル（HTML/CSS/画像）の高速配信、リクエストの受付・振り分け | Java で動くJSP/Servletの実行、複雑な処理ロジック      |
| 扱うデータ   | 生のHTTPリクエスト（テキスト）                                      | 扱いやすい Java オブジェクト（`request`, `response`） |
| 研修での扱い | Tomcat に内蔵されている機能で代用                                   | Tomcat が主役                                         |

小規模な学習環境では **Tomcat 単体で両方の役割**を担ってしまいます。本研修でもこのパターンです。ただし、本格的な業務システムでは前段に Apache や Nginx を置いて、静的配信や負荷分散を任せるのが一般的です。

#### 📦 request オブジェクトとは何か

Tomcat は、ブラウザから届いた生のHTTPリクエスト（テキストの塊）を解析して、**`request` という Java オブジェクト**に整形してくれます。このおかげで、プログラマーは生のテキストを自分で解析する必要がなく、`request.getParameter("username")` のようなメソッドを呼ぶだけで入力値を取り出せます。

［図（テキスト抽出）：🔤 生のHTTPリクエスト / （Webサーバーが受け取る形） / POST /training/greet.jsp HTTP/1.1 / Host: localhost:8080 / Content-Type: / application/x-www-form-urlencoded / Content-Length: 22 / username=田中&age=25 / ↑ 自分で解析するのは面倒 / Tomcat が / 解析・整形 / 📦 request オブジェクト / （JSP/Servletが受け取る形） / request.getParameter("username") / → "田中" / request.getParameter("age") / → "25" / request.getMethod() / → "POST" / メソッドを呼ぶだけで取り出せる］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 9-2：Tomcat が生のHTTPリクエストを扱いやすい request オブジェクトに変換

#### 📤 response の役割（HTML を返す側）

`request` の対になるのが `response` オブジェクトです。こちらは**ブラウザへ返すデータを組み立てる側**です。ただしJSPでは、ファイル内に書いたHTMLがそのまま response に書き込まれるため、プログラマが直接 response を意識する場面はあまりありません。

> **🔑 2つのオブジェクトの関係**
>
> - **request** ─ 入口。ブラウザからの入力を「読む」ために使う
> - **response** ─ 出口。ブラウザへの出力を「書く」ために使う
>
> JSPでは `request.getParameter()` などで読み取り、ページに書いたHTML自体が自動で response に書き込まれていきます。

#### 🔀 複雑な処理 → JSPへ渡す：forward パターン

ここまでは「JSPで入力を受け取ってそのままHTMLを返す」形を想定してきました。ただ、実際のアプリケーションでは**処理とHTML生成を分けた方が見通しがよい**場面が増えます。

その時の定番パターンが、DAY 4 で詳しく学ぶ **MVC** ─ 処理担当の Servlet が計算し、結果をHTMLとして整形するのは JSP に任せる、という役割分担です。

［図（テキスト抽出）：🌐 ブラウザ / form送信 / ⚙️ Servlet（処理役） / 1. request から入力取得 / 2. 計算・DB検索など / 3. 結果を request に詰める / setAttribute("result", ...) / forward / （request を渡す） / 📄 JSP（表示役） / 1. request から結果取得 / 2. HTMLを response に書き込む / getAttribute("result") / HTML / 🌐 / 表示 / forward：処理は Servlet、表示は JSP / Servlet は計算結果を request に「詰めて」JSP に渡し、 / JSP はその結果を取り出して HTML を response に書き込んで返す / 📥 request オブジェクト / 入口：Servlet→JSPの橋渡し役 …］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 9-3：forward パターン ─ Servlet は request に結果を詰め、JSP は response に HTML を書き込む

#### ✍️ JSP が書いた HTML はどこへ行く？

「JSPが response に書き込む」と言われても、実際にコードを書くときに `response.write(...)` のようなメソッドを呼んでいる場面はありません。これには理由があります ─ **JSP ファイルに書いた HTML テキストは、自動的に response オブジェクトの出力ストリームに流し込まれている**からです。

［図（テキスト抽出）：JSPの各行が response に書き込まれる流れ / 📄 greet.jsp（ソース） / \<html\>\<body\> / \<% String name = request.getParameter("u"); %\> / \<h1\>こんにちは、 / \<%= name %\> / さん！\</h1\> / \</body\>\</html\> / HTML（そのまま出力） / Javaコード（実行される） / 📤 response の出力内容 / （ブラウザへ届くHTML） / \<html\>\<body\> / \<h1\>こんにちは、 / 田中 / ← 値に置換 / さん！\</h1\> / \</body\>\</html\> / この内容がそのままブラウザに送られる / ※ \<% ... %\> の Java コード自体は出力されない / JSPはTomcatが内部でServletに変換され、その中で response. …］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 9-4：JSPの各行は Tomcat によって response に書き込まれる

> **📌 JSPの仕組み ─ 裏側では Servlet になっている**
>
> JSPファイルは、実行時に Tomcat によって**自動的に Servlet クラスに変換**されます。変換後のコードでは、HTML部分が `out.write("<h1>...")` のような呼び出しに置き換えられ、それが最終的に response の出力ストリームへ流し込まれます。  
> つまり「JSPに書いた文字列 = response に書き込まれる内容」と考えて問題ありません。

**JSPが内部で変換される様子（イメージ）**

    // 元のJSP
    <h1>こんにちは、<%= name %>さん！</h1>

    // ↓ Tomcatが内部で自動変換

    out.write("<h1>こんにちは、");
    out.write(name);   // <%= %> の部分
    out.write("さん！</h1>");

    // ↑ この out は response の出力先と繋がっている
    //   response.getWriter() が内部で使われる

> **💡 なぜJSPがHTML生成を担当するのか**
>
> JSPの得意分野は **HTMLの中にJava式を埋め込む**こと。逆に Servlet で `out.println("<html>...</html>")` と大量に書くのは読みづらく保守しにくい。そこで「**処理はJava（Servlet）で、HTML出力はテンプレート（JSP）で**」という役割分担が生まれました。DAY 4 の MVC パターンで本格的に使います。

Webアプリの処理を「野球やサッカーのボール回し」に例えると、**今ボールを持っているのは誰か＝今どのプレイヤーが処理を進めているか**がわかりやすくなります。

### 9-3. 実践：名前を受け取って挨拶するページ

**greet.html（入力フォーム）**

```html
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <title>挨拶フォーム</title>
</head>
<body>
  <h1>名前を入力してください</h1>
  <form action="greet.jsp" method="post">
    <label>名前：<input type="text" name="username"></label>
    <button type="submit">送信</button>
  </form>
</body>
</html>
```

**greet.jsp（受信側）**

```jsp
<%@ page contentType="text/html; charset=UTF-8" %>
<%
  // 文字化け対策
  request.setCharacterEncoding("UTF-8");

  // フォームから送られた値を受け取る
  String username = request.getParameter("username");
%>
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <title>挨拶</title>
</head>
<body>
  <% if (username != null && !username.isEmpty()) { %>
    <h1>こんにちは、<%= username %>さん！</h1>
  <% } else { %>
    <h1>名前が入力されていません</h1>
  <% } %>
  <p><a href="greet.html">戻る</a></p>
</body>
</html>
```

> **⚠️ 文字化け対策**
>
> 日本語のフォームデータを受け取る場合、`request.setCharacterEncoding("UTF-8")` を `getParameter()` の前に書くことが重要です。これを忘れると文字化けします。

### 9-4. GETとPOSTの違い

|               | GET                       | POST                 |
|---------------|---------------------------|----------------------|
| データの場所  | URLの末尾（?name=value）  | HTTPリクエストの本文 |
| URLに見える？ | 見える                    | 見えない             |
| データ量      | 少量（URL長の制限あり）   | 大量OK               |
| 用途          | 検索、フィルタ            | 登録、ログイン、更新 |
| ブックマーク  | 可能（URLに含まれるから） | 不可能               |

#### 🎯 GET は主に「制御情報」を渡すために使う

GETでよく送る情報は、**ユーザーが入力したデータ**というより、**「画面の表示の仕方を指定する情報」**です。典型的な使い方：

| 用途           | URLの例                       | 何を伝えているか         |
|----------------|-------------------------------|--------------------------|
| ページ指定     | `/products?page=2`            | 一覧の2ページ目を見せて  |
| 表示モード切替 | `/article?mode=edit`          | 編集モードで開いて       |
| 並び替え       | `/list?sort=name&order=asc`   | 名前順で昇順に並べて     |
| 検索条件       | `/search?q=田中&category=dev` | 田中さんを開発部から検索 |
| 個別ID指定     | `/user?id=123`                | ID 123 番の詳細を表示    |

> **📌 GETとPOSTの使い分けの目安**
>
> - **GET**：どれを／どう表示するかの**制御情報**（ページ番号、並び順、フィルタ条件、ID）
> - **POST**：ユーザーが**入力したデータ**（名前、メール、パスワード、メッセージ）
>
> GETで送る情報は「URLに含まれて残っても困らない情報」、POSTで送る情報は「URLに残したくない／長すぎる情報」と考えると自然です。

#### 🔧 GET は URL を直接叩いてテストできる

GETの大きな利点は、**ブラウザのアドレスバーから直接テストできる**ことです。

例：SearchServlet を `@WebServlet("/search")` で `doGet` 対応にしておけば、次のURLをアドレスバーに入力するだけで検索を実行できます：

**ブラウザのアドレスバーに直接入力してテスト**

    http://localhost:8080/training/search?keyword=田中
    http://localhost:8080/training/products?page=3&sort=price
    http://localhost:8080/training/user?id=123

フォームHTMLを用意しなくても動作確認できるので、**開発初期のデバッグに便利**です。URLだけで再現できるので、チーム内で「このバグが出る」と共有するのも簡単です。

> **🔍 POSTのテストには工夫が必要**
>
> POSTリクエストはアドレスバーからは送れません。テストするには以下のいずれかが必要です：
>
> - 実際にフォームを作って画面から送信する（研修ではこの方法）
> - `curl` コマンドで送信：  
>   `curl -X POST http://localhost:8080/training/register -d "name=田中&email=a@b.c"`
> - Postman などの API テストツールを使う
> - ブラウザの開発者ツールで手動送信する

### 9-5. アクセスログから見る GET と POST の決定的な違い

Webサーバーは、届いた全リクエストを**アクセスログ**に記録しています。Tomcat ではデフォルトでログ出力が有効になっており、リクエストの記録が自動的に残ります。このログを実際に見ると、GET と POST の違いがはっきり分かります。

Tomcat 起動後、以下のファイルが日付ごとに自動生成されます。

**アクセスログのパス**

    C:\apache-tomcat-10.1.x\logs\localhost_access_log.YYYY-MM-DD.txt

#### 🔐 セキュリティ上の重要な意味

|                                    | GET（ハガキ）                | POST（封筒）               |
|------------------------------------|------------------------------|----------------------------|
| パラメータがログに残る？           | **残る**                     | **残らない**               |
| サーバー管理者から見える？         | 見える（ログを開けば誰でも） | 見えない（通常のログでは） |
| ブラウザ履歴に残る？               | 残る（URLごと）              | 残らない                   |
| Referer ヘッダで他サイトに漏れる？ | 漏れる可能性あり             | 漏れない                   |
| パスワード送信に使ってよい？       | ❌ **絶対にダメ**            | ✓ こちらを使う             |

**🚨 実例：GETでパスワードを送ると何が起きるか**

もし悪い設計のログイン画面が `GET` でパスワードを送ったとします：

```
GET /login?id=yamada&password=mysecret123 HTTP/1.1
```

この一行のリクエストが、以下の**5箇所以上**に記録・露出します：

1.  **サーバーのアクセスログ** ─ 平文でファイルに永久保存
2.  **プロキシサーバーのログ** ─ 経路上のすべての中継機器に記録される可能性
3.  **ブラウザの履歴** ─ アドレスバーに表示されたものすべて
4.  **Referer ヘッダ** ─ そのページから外部リンクをクリックすると、リンク先のサイトにURLが送られる
5.  **ブックマーク・URL共有** ─ ユーザーがURLをコピペしてSlackやメールに貼ると拡散

→ **サーバー管理者・ネットワーク管理者・他社サイト・友人にまで**、パスワードが漏れていく可能性があります。

> **💡 ログ漏洩は実際に起きている**
>
> 「サーバーログにパスワードが書かれていた」という事故は実際に多数報告されています。代表的なパターン：
>
> - 開発者がデバッグ用に GET でパスワードを送る実装にしてしまい、本番に残ったまま
> - サーバーログが圧縮されてバックアップサーバーに転送 → そのサーバーが侵害される
> - クラウドのログ集約サービスにログを送信 → 第三者サービスにも履歴が残る
> - 外部委託先のログ解析業者にログを渡す → 平文パスワードが社外に渡る
>
> これらは**「設計時にGETを使ってしまった」だけで起きる事故**です。POSTを使っていれば防げました。

> **📌 業界標準でも「GETで秘密情報を送るな」と明記**
>
> OAuth 2.0 の認証トークンを規定する **RFC 6750**（IETF が策定）にも、はっきりとこう書かれています：
>
> 「Bearer tokens SHOULD NOT be passed in page URLs ... attackers might be able to steal them from the history data, logs, or other unsecured locations.」  
> （認証トークンは URL に含めるべきでない。攻撃者が履歴・ログ・その他の保護されていない場所から盗み出せる可能性があるため）
>
> これは認証トークンの話ですが、**パスワード・個人情報・APIキーなど機密情報すべてに当てはまる原則**です。

#### 📊 GETでも気をつけるべきプライバシー

パスワードほど深刻でなくても、GETで送る情報は**「サーバー管理者に見られても困らないか」**を考える必要があります。

| 例                   | GETで送ってOK？ | 理由                                                           |
|----------------------|-----------------|----------------------------------------------------------------|
| 商品検索キーワード   | △ 一般的にはOK  | ただし「妊娠検査薬」「うつ病」等の検索は履歴に残ると気持ち悪い |
| ページ番号、並び順   | ○ 問題なし      | 制御情報なので残っても困らない                                 |
| ユーザーID（氏名）   | △ 注意          | 誰がいつアクセスしたかが追跡可能になる                         |
| クレジットカード番号 | ❌ 絶対NG       | 金融情報は最高機密                                             |
| 位置情報（緯度経度） | △ 注意          | 個人の行動追跡につながる                                       |

>

#### 🎫 トークンとは何か

トークンは**「あなたが認証済みであることを示すデジタルの証明書」**のような文字列です。それ自体は単なる長い文字列です。

**トークン本体の例（JWT形式・ヘッダー.ペイロード.署名の3パートを「.」で連結）**

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IllhbWFkYSIsImlhdCI6MTcxNTI0NTYyMn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

- **ランダム（または暗号学的に署名された）長い文字列**
- これを持っている＝サーバーが「認証済みユーザー」と認める
- 代表的な形式：**JWT**（JSON Web Token）、不透明トークン、APIキー など
- JWT の場合、ピリオド（`.`）で 3 つのパートに分かれている（ヘッダー・ペイロード・署名）

#### 📨 トークンは「HTTPヘッダー」に乗せて送る

HTTPリクエストは、Ch.9-1 で見たように**ヘッダー（先頭部分）**と**ボディ（本文）**から成ります。トークンは標準的には、ヘッダーの中の `Authorization` という名前のヘッダーに乗せて送ります。

［図（テキスト抽出）：HTTPリクエスト全体の中でのトークンの位置 / 📨 HTTPリクエスト全体 / POST /api/users HTTP/1.1 / ↑ リクエストライン（メソッド・パス・HTTPバージョン） / ▼ ヘッダー部分（複数のヘッダーが並ぶ） / Host: api.example.com / Content-Type: application/json / Authorization: / Bearer eyJhbGciOiJIUzI1NiIs... / User-Agent: Mozilla/5.0... / ▼ ボディ部分 / {"name": "山田", "email": "..."} / ← ★ ここ］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 9-9：トークンは Authorization という名前のヘッダーに乗せて送る

#### 🔍 Authorization ヘッダーの構造

`Authorization` ヘッダーは、**「認証スキーム名」**と**「認証情報（トークン本体）」**を半角スペースで区切って書きます。

**Authorization ヘッダーの書式**

    Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5...
    ↑ヘッダー名    ↑スキーム ↑トークン本体

| 要素            | 内容           | 説明                                                       |
|-----------------|----------------|------------------------------------------------------------|
| `Authorization` | HTTPヘッダー名 | 「認証情報を運ぶ専用のヘッダー」と HTTP 仕様で決まっている |
| `Bearer`        | 認証スキーム名 | 「Bearer トークン」を扱う標準スキーム（RFC 6750）          |
| `eyJhbGc...`    | トークン本体   | 9-6 冒頭で見た長い文字列。これだけが本来の「トークン」     |

**💡 認証スキームは Bearer 以外にもある**

`Authorization` ヘッダーで使える認証スキームは Bearer 以外にもあります。代表的なものを紹介します：

| スキーム | 用途                                    | 例                                          |
|----------|-----------------------------------------|---------------------------------------------|
| `Bearer` | OAuth 2.0 / JWT などのトークン          | `Authorization: Bearer eyJhbGc...`          |
| `Basic`  | ID/PWをBase64エンコードして送る古い方式 | `Authorization: Basic dXNlcjpwYXNz`         |
| `Digest` | パスワードをハッシュ化して送る方式      | `Authorization: Digest username="...", ...` |

現代のWebアプリ・APIでは `Bearer` が事実上の標準です。

> **📌 「トークン」と「Authorization ヘッダー」の関係を整理**
>
> - **トークン本体**は単なる文字列（例：`eyJhbGc...`）
> - その文字列を運ぶ**入れ物**が `Authorization` という HTTP ヘッダー
> - ヘッダーの中では `Bearer` という**スキーム名**を付けて「これは Bearer トークンだ」と明示する
> - 図でいえば「トークン本体」は**中身**、「Authorization ヘッダー」は**封筒**、「Bearer」は**封筒に書く宛名種別**のような関係

#### ⏰ トークンには有効期間がある

トークンが永久に有効だと、漏洩したときに被害が無限大になります。そこで**有効期間を設けて**、定期的に更新する仕組みになっています。

| 種類                     | 有効期間             | 役割                                         |
|--------------------------|----------------------|----------------------------------------------|
| **アクセストークン**     | 数分〜数時間（短命） | 普段のAPI呼び出しに使う                      |
| **リフレッシュトークン** | 数日〜数週間         | アクセストークン期限切れ時に新しいものを取得 |

> **💡 短い有効期間のメリット**
>
> - 万一漏洩しても、攻撃者が使える時間が限られる
> - 期限切れ後は自動で無効になり、被害が拡大しない
> - 不正利用の検知後、トークンを失効させるだけで対応できる
>
> ユーザーは普段、有効期限を意識しません。アプリが裏でリフレッシュトークンを使って自動更新しているからです。

#### 📤 トークンの送り方：標準・限定OK・絶対ダメ

トークンの送信方法には3種類あります。それぞれセキュリティレベルが違います。

［図（テキスト抽出）：トークンの送信方法 ─ 安全性で色分け / ✅ ① Authorization ヘッダー（標準・推奨） / Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5... / RFC 6750 で標準化／ログ・履歴に残らない／HTTPの専用ヘッダー / ⚠️ ② POST のリクエストボディ（限定的にOK） / POST /api/data HTTP/1.1 / access_token=eyJhbGc... / ヘッダーが使えない場合のみ／ログには残らないがヘッダーが第一選択 / ❌ ③ URL のクエリパラメータ（絶対ダメ） / GET /api/data?access_token=eyJhbGc... HTTP/1.1 / → ログ・履歴・Referer・URL共有 に残る！攻撃者に盗まれてアカウント乗っ取り］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 9-10：トークンの送信方法は3種類。Authorization ヘッダーが標準

#### 📋 なぜ Authorization ヘッダーが標準なのか

| 項目                    | Authorization ヘッダー | URL クエリパラメータ           |
|-------------------------|------------------------|--------------------------------|
| HTTPの仕様での位置づけ  | 認証情報専用の場所     | リソース指定用（流用は本来NG） |
| アクセスログに記録？    | 通常記録されない       | 記録される                     |
| ブラウザ履歴に残る？    | 残らない               | 残る                           |
| Refererヘッダで漏れる？ | 漏れない               | 漏れる可能性                   |
| URL共有で漏れる？       | 漏れない               | 漏れる                         |

> **🚨 URLクエリパラメータでトークンを送ると何が起きるか**
>
> 9-5 で見たパスワードと同じく、URL に含めたトークンは複数箇所に漏洩します。トークンが漏れた場合の被害は**有効期間中の認証済み状態の乗っ取り**：
>
> - 攻撃者がそのトークンを使って、本人になりすましてAPI を呼び出せる
> - 有効期限まで（数分〜数時間）、好きなだけアカウントを操作できる
> - 「読み取り権限」のトークンでも、個人情報の閲覧は可能
> - 「書き込み権限」のトークンなら、データの改ざん・削除も可能

> **📌 RFC 6750 の公式な警告**
>
> OAuth 2.0 のトークン使用法を規定する **RFC 6750**（IETF 標準）には、次の通り明記されています：
>
> 「Bearer tokens SHOULD NOT be passed in page URLs ... attackers might be able to steal them from the history data, logs, or other unsecured locations.」
>
> （認証トークンは URL に含めるべきでない。攻撃者が履歴・ログ・その他の保護されていない場所から盗み出せる可能性があるため）

#### 🚀 研修との関係

本研修では、トークン認証は実装しません。しかし、**現代のWebアプリ・APIではほぼ必須**の仕組みです。次のステップで以下を学べると、この内容と繋がります：

| 技術                       | 位置づけ                                                                           |
|----------------------------|------------------------------------------------------------------------------------|
| `HttpSession`              | 本研修の Servlet で使えるセッション管理（Cookie ベース。広い意味でトークンの一種） |
| JWT（JSON Web Token）      | Java 用ライブラリで実装可能。署名付きの自己完結型トークン                          |
| OAuth 2.0 / OpenID Connect | Google・GitHub 認証など外部認証連携の標準                                          |
| Spring Security            | Spring Boot での認証実装の標準ライブラリ                                           |

> **🔑 まとめ：トークンと GET/POST の関係**
>
> - 現代のWeb認証は**事前にトークンを発行**して、以降はそれで認証する
> - トークンには**有効期間**があり、漏洩しても被害が限定的
> - トークンの送信は **Authorization ヘッダー**（POST のボディ も限定的にOK）
> - URL に含めると 9-5 で見た通り**ログ・履歴・Refererに漏れる**のでダメ
> - これが「URLには制御情報、機密情報は POST やヘッダーに」という原則の核心

### 13-1. Flexbox の主要プロパティまとめ

| 対象            | プロパティ        | 役割                            |
|-----------------|-------------------|---------------------------------|
| 親（container） | `display: flex`   | Flexboxを有効にする             |
|                 | `flex-direction`  | 並びの方向（row / column）      |
|                 | `justify-content` | 主軸方向の配置                  |

|                 | `flex-shrink`     | 縮む時どれだけ縮むか            |
|                 | `flex-basis`      | 基本サイズ                      |

> **📌 flex: 1 1 280px の意味**

DAY 3 ─ Chapter 14 ⏱ 1.5h CSS Grid 2次元レイアウトの決定版 ─ カード並びを Grid に書き換える

### 14-1. FlexboxとGridの使い分け

|            | Flexbox                 | Grid                             |
|------------|-------------------------|----------------------------------|
| 得意なこと | 1方向の並び（横 or 縦） | 2方向（行と列の同時制御）        |
| 向く場面   | ナビバー、ツールバー    | カード一覧、ダッシュボード       |
| 考え方     | 中身が並び方を決める    | 枠（グリッド線）が並び方を決める |

### 14-2. Gridの基本記法

**Gridの基本**

    .container {
      display: grid;
      grid-template-columns: repeat(3, 1fr);  /* 3等分 */
      gap: 16px;
    }

    /* 自動調整版：画面幅に応じて列数が変わる */
    .container {
      grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
    }

### 15-3. メディアクエリの書き方

メディアクエリは、CSSの中に「条件付きのブロック」を書くための仕組みです。`@media` で始まり、条件を満たしたときだけ中のCSSが適用されます。

［図（テキスト抽出）：メディアクエリの基本構造 / @media / ( / max-width / : / 768px / ) { / .card-list { / grid-template-columns / : / 1fr / ; / } / ① 宣言 / ② 条件（画面幅 768px 以下） / ③ 適用するCSS］（元HTMLのSVG図。Markdown版では図中テキストのみ掲載）

図 15-2：メディアクエリは「宣言 + 条件 + CSS」の3要素

#### 📝 よく使う条件の書き方

| 条件                                        | 意味                          | 使い道                                         |
|---------------------------------------------|-------------------------------|------------------------------------------------|
| `(max-width: 768px)`                        | 画面幅が 768px **以下**のとき | 大きい画面で作って、小さい画面用に調整（主流） |
| `(min-width: 769px)`                        | 画面幅が 769px **以上**のとき | モバイルファーストで作って、大画面用に拡張     |
| `(min-width: 480px) and (max-width: 768px)` | 480px 以上 かつ 768px 以下    | タブレットだけに適用したいとき                 |
| `(orientation: portrait)`                   | 縦向き画面のとき              | スマホの回転対応                               |

> **💡 max-width と min-width の使い分け**
>
> 本研修では `max-width` で統一しています（PCのスタイルを基本として、狭くなったときに上書きする方式）。逆に `min-width` を使うと「まずスマホ用を書いて、広くなったら上書き」する「モバイルファースト」の書き方になります。どちらも正解ですが、混ぜると混乱するので1つに統一するのがコツです。

#### ⚙️ メディアクエリの動作原理

ブラウザは常に画面幅を監視していて、幅が変わるたびに「条件に当てはまるメディアクエリ」を判定します。条件を満たせばそのブロック内のCSSが適用され、満たさなければ無視されます。**ウィンドウをリサイズすると、リロードなしで即座に適用/解除が切り替わる**のはこのためです。

**🔑 メディアクエリは「後から上書き」の仕組み**

メディアクエリの中で書いたCSSは、**外側で定義した同じプロパティを上書き**します。だからメディアクエリのブロックは通常、CSSファイルの**最後にまとめて書きます**（CSSは後に書いたルールが優先されるため）。

```
/* 基本スタイル（PC想定） */
.card-list { grid-template-columns: repeat(3, 1fr); }

/* ↓ メディアクエリは最後 */
@media (max-width: 768px) {
  .card-list { grid-template-columns: 1fr; } /* 上書き */
}
```

DAY 4 ─ Chapter 17 ⏱ 2.0h HttpServletの基本 Servletの書き方をマスターする

### 17-1. はじめてのServlet

**HelloServlet.java**

```java
import java.io.*;
import jakarta.servlet.*;
import jakarta.servlet.http.*;
import jakarta.servlet.annotation.*;

// ① URLパターンを指定（ブラウザからこのURLでアクセスできる）
@WebServlet("/hello")
public class HelloServlet extends HttpServlet {

    // ② GETリクエスト時に呼ばれるメソッド
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // ③ レスポンスの文字コードを設定
        response.setContentType("text/html; charset=UTF-8");

        // ④ HTMLを出力
        PrintWriter out = response.getWriter();
        out.println("<html><body>");
        out.println("<h1>Hello, Servlet!</h1>");
        out.println("</body></html>");
    }
}
```

### 17-2. @WebServlet アノテーション

Servlet 3.0以降では、`@WebServlet` アノテーションでURLパターンを指定できます。web.xmlに書く従来の方法より簡潔です。

| 属性                    | 意味                    | 例                      |
|-------------------------|-------------------------|-------------------------|
| `value` / `urlPatterns` | マッピングするURL       | `"/hello"`, `"/user/*"` |
| `name`                  | Servletの名前（省略可） | `"HelloServlet"`        |

### 17-3. doGet と doPost

| メソッド   | 呼ばれるタイミング | 用途                     |
|------------|--------------------|--------------------------|
| `doGet()`  | GETリクエスト時    | ページ表示、検索         |
| `doPost()` | POSTリクエスト時   | フォーム送信、データ更新 |

**doPostの例**

    @WebServlet("/greet")
    public class GreetServlet extends HttpServlet {

        @Override
        protected void doPost(HttpServletRequest request,
                              HttpServletResponse response)
                throws ServletException, IOException {

            request.setCharacterEncoding("UTF-8");
            String name = request.getParameter("username");

            // 処理結果をリクエスト属性にセット
            request.setAttribute("greeting", "こんにちは、" + name + "さん！");

            // JSPに転送（forward）
            RequestDispatcher rd = request.getRequestDispatcher("/WEB-INF/jsp/greet.jsp");
            rd.forward(request, response);
        }
    }

> **📌 forward とは**
>
> `forward` はサーバー内部でリクエストを別のリソース（JSP等）に引き渡す仕組みです。ブラウザのURLは変わりません。Servletで処理した結果を `request.setAttribute()` で渡し、JSPで表示する ─ これがServlet + JSP連携の基本パターンです。

DAY 4 ─ Chapter 18 ⏱ 1.5h リクエストとレスポンス処理 HttpServletRequest / HttpServletResponse を使いこなす

### 18-1. HttpServletRequest の主なメソッド

| メソッド                    | 戻り値     | 用途                       |
|-----------------------------|------------|----------------------------|
| `getParameter("key")`       | String     | フォームの入力値を取得     |
| `getParameterValues("key")` | String\[\] | チェックボックス等の複数値 |
| `setAttribute("key", obj)`  | void       | JSPに渡すデータをセット    |
| `getAttribute("key")`       | Object     | セットされたデータを取得   |
| `getMethod()`               | String     | "GET" or "POST"            |
| `getRequestURI()`           | String     | リクエストされたURI        |
| `setCharacterEncoding()`    | void       | 文字コードの指定           |

### 18-2. HttpServletResponse の主なメソッド

| メソッド                                     | 用途                          |
|----------------------------------------------|-------------------------------|
| `setContentType("text/html; charset=UTF-8")` | レスポンスの形式と文字コード  |
| `getWriter()`                                | HTML出力用のPrintWriterを取得 |
| `sendRedirect("url")`                        | 別URLにリダイレクト           |
| `setStatus(404)`                             | HTTPステータスコードを設定    |

### 18-3. forward と redirect の違い

|                | forward（転送）             | redirect（リダイレクト）        |
|----------------|-----------------------------|---------------------------------|
| URLバー        | 変わらない                  | 変わる                          |
| 処理の場所     | サーバー内部                | ブラウザに「別URLに行け」と指示 |
| データの受渡し | request.setAttribute で可能 | 不可（別リクエストになる）      |
| 用途           | Servlet → JSP で結果表示    | 処理完了後に別ページへ移動      |

**forward vs redirect の使い分け**

    // forward：処理結果をJSPに渡して表示する場合
    request.setAttribute("data", someData);
    request.getRequestDispatcher("/WEB-INF/jsp/result.jsp")
           .forward(request, response);

    // redirect：処理完了後に別のページに飛ばす場合
    // （例：ログイン成功後にトップページへ）
    response.sendRedirect("top");

**SearchServlet.java**

```java
@WebServlet("/search")
public class SearchServlet extends HttpServlet {

    // 仮の社員データ（実務ではDBから取得）
    private static final String[][] EMPLOYEES = {
        {"田中太郎", "開発部", "tanaka@example.com"},
        {"佐藤花子", "営業部", "sato@example.com"},
        {"鈴木一郎", "人事部", "suzuki@example.com"},
        {"田中美咲", "経理部", "tanaka.m@example.com"}
    };

    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");
        String keyword = request.getParameter("keyword");

        // 検索ロジック
        java.util.ArrayList<String[]> results = new java.util.ArrayList<>();
        if (keyword != null && !keyword.trim().isEmpty()) {
            for (String[] emp : EMPLOYEES) {
                if (emp[0].contains(keyword)) {
                    results.add(emp);
                }
            }
        }

        // 結果をリクエスト属性にセットしてJSPに転送
        request.setAttribute("keyword", keyword);
        request.setAttribute("results", results);
        request.getRequestDispatcher("/WEB-INF/jsp/searchResult.jsp")
               .forward(request, response);
    }
}
```

> **📌 setAttribute / getAttribute の流れ**
>
> Servlet側で `request.setAttribute("results", results)` → JSP側で `request.getAttribute("results")` で取り出す。これが Servlet → JSP のデータ受け渡しの基本です。取り出す際は `Object` 型で返るのでキャストが必要です。

```java
@WebServlet("/bmi")
public class BmiServlet extends HttpServlet {
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");
        double height = Double.parseDouble(
            request.getParameter("height")) / 100.0; // cm → m
        double weight = Double.parseDouble(
            request.getParameter("weight"));
        double bmi = weight / (height * height);

        String category;
        if (bmi < 18.5) category = "やせ型";
        else if (bmi < 25.0) category = "標準";
        else category = "肥満";

        request.setAttribute("bmi", String.format("%.1f", bmi));
        request.setAttribute("category", category);
        request.getRequestDispatcher("/WEB-INF/jsp/bmiResult.jsp")
               .forward(request, response);
    }
}
```

>
> - データベース（MySQL / PostgreSQL 等）の接続は本研修の範囲外
> - 今回の学習目的は「入力 → 処理 → 表示」の **MVCの流れを掴むこと**

> **🚀 応用チャレンジ（余裕のある人向け）**
>
> DB なしでも「登録」を体験してみたい方へ、以下のステップで拡張できます：

> 2.  確定 Servlet でデータを `ArrayList`（メモリ上のリスト）に追加
> 3.  一覧ページ（`/register/list`）で保存された内容を表示
>

```java
@WebServlet("/register")
public class RegisterServlet extends HttpServlet {
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");

        String name = request.getParameter("name");
        String dept = request.getParameter("department");
        String email = request.getParameter("email");

        // バリデーション
        java.util.ArrayList<String> errors = new java.util.ArrayList<>();
        if (name == null || name.trim().isEmpty())
            errors.add("名前は必須です");
        if (dept == null || dept.isEmpty())
            errors.add("部署を選択してください");
        if (email == null || email.trim().isEmpty())
            errors.add("メールアドレスを入力してください");

        // データをセット
        request.setAttribute("name", name);
        request.setAttribute("department", dept);
        request.setAttribute("email", email);
        request.setAttribute("errors", errors);

        // JSPに転送
        request.getRequestDispatcher("/WEB-INF/jsp/confirm.jsp")
               .forward(request, response);
    }
}
```

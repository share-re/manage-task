## 1. 背景・ねらい

現状の AI内田さんは「内田さんの口調で答える AI」だが、次の弱点がある。

- **見た目がキャラクターとして立っていない**（アイコンに表情の仕組みはあるが、小さくて存在感が弱い）
- **社内の知識を持っていない**。研修で語られた内容や内田さん本人の考えを知らないため、
  答えが「どこにでもある一般論」に留まる
- そのため「内田さんに相談している」感が薄く、**相談窓口として選ばれにくい**

クエストカードの要望は次のとおり。

> もっと人間に寄せて親しみやすく。研修や内田さんの情報を吸い上げ、
> 一般論から内田さんの論まで相談できるように

これを実現し、AI内田さんを「冷たい神託」ではなく「隣に座る師匠」にする。

## 3. 用語（今回はじめて出るもの）

| 用語 | 意味 |

## 5. Q2：研修・内田さん情報のナレッジ化（RAG）

今回いちばん重い条件。ここが Q3・Q4 の土台になる。

> 補足（2026-07-22 方式変更）：検索方式を**ベクトル検索 → キーワード検索（語句一致）**に変更した。
> 理由は、期限に対して埋め込み API の1日上限（無料枠 1,000回/日）が取り込み・テストの制約になるため。
> キーワード検索は埋め込みを一切使わず、この上限と無関係になる。代償として言い換え・同義語に弱くなる
> （例：「責任」という語では「あなたの成果物」の節を引けない）。ベクトル検索は時間に大きな余裕があれば
> 後乗せする（§2.1。実装済みの埋め込み経路は温存）。

### FR-Q2-01 知識の元ネタ

| 元ネタ | 状態 |
| --- | --- |
| 研修の資料・議事録 | 数百ページ以上。**形式は Markdown / テキスト**に統一する |
| 内田さん本人へのインタビュー | `docs/uchida-interview-sheet.md` が既にある。今後追加取材も行う |

- **PDF / PowerPoint / Word は対象外**とし、登録前に Markdown へ変換しておく運用とする。
  （変換処理を作らずに済み、開発量を大きく減らせるため）
- Markdown への変換と、機密が含まれないことの確認（NFR-Q2-01）は、**資料を登録する本人が行う**。
  この手作業は今回のクエストの作業量に含めて見積もる（数百ページ規模のため軽くはない）。

### FR-Q2-02 知識の置き場所と登録方法

- 資料はリポジトリ内（例：`docs/knowledge/`）に Markdown ファイルとして置き、**PR 経由で更新**する。
- 資料を追加・変更したら、**手動でコマンドを実行**して取り込み処理を走らせる（例：`npm run ingest`）。
  - 自動化（CI での実行）は今回行わない。**何が起きているか見えやすくするため**、まず手動から始める。
- 取り込みは**ファイル名（`source_file`）をキー**に行う。ある資料の行はファイル名に紐づき、再実行時は
  「そのファイル名の行を消して現在の中身で入れ直す」。**中身そのものによる名寄せ・重複排除はしない。**
- 取り込み処理は**同期型**とする：実行後の `knowledge` の中身が、そのとき `docs/knowledge/` にある資料の内容と
  **一致する**こと。具体的には次を満たす。
  - **冪等（べきとう）**：ファイル名も中身も変わっていなければ、何回実行しても行は同一で、増えない。
    判定：ingest を**2回連続**で実行し、1回目の後と2回目の後で**行数・各行の内容が完全に一致**すればOK
  - **修正**：ファイル名が同じで中身が変わったら、そのファイル名の行が新しい内容に置き換わる
  - **削除／改名**：ファイルが無くなれば旧名の行は消える（改名は「旧名の削除＋新名の新規取り込み」として扱われる）
  - **割り切り**：同じ中身を**別ファイル名**で置くと、内容は重複して入る（**名前が違えば別資料扱い**。中身での名寄せはしない）

### FR-Q2-03 知識の保存先

- Supabase に**新規テーブル**を作る（例：`knowledge`）。既存の `conversations` / `messages` とは
  **役割が異なるため別テーブル**とする。

| テーブル | 中身 | 持ち主 | 用途 |
| --- | --- | --- | --- |
| `conversations` / `messages`（既存） | 利用者と AI のやりとり | ユーザーごと（本人のみ閲覧） | 続きから相談する |
| `knowledge`（新規） | 資料のチャンク＋ベクトル | 全員共通 | AI が答えの根拠として検索する |

- 検索は**サーバ側（API）**が行う。RLS（＝行単位アクセス制御。行ごとに誰が読めるか決める仕組み）の
  「ログイン済みは**読み取りのみ**」は多層防御の保険（万一クライアントから触られても書けない）。書き込みは取り込み処理のみ。
- 列定義は実装で確定：`id / source_file / heading / chunk_index / content / embedding（vector 768次元）/ created_at`
  （`scripts/sql/knowledge.sql`）。`embedding` 列は**ベクトル後乗せ用**で、キーワード検索では未使用（空でよい）。

### FR-Q2-04 検索と回答生成

1. ユーザーの発言を受け取る
2. 発言から**語句（キーワード）を取り出し**、`knowledge` の見出し・本文に**その語を含むチャンク**を探す
3. **一致した語数が基準以上のチャンク**（一致数の多い順・最大4件）を人格プロンプトに添えて Gemini に渡す
4. 基準を満たすチャンクが1件も無ければ、**何も添えずに従来どおり応答する**

- 「添えない」判断は**一致語数による足切り**で実現する（初期値：2語以上一致。質問から取れた語が
  1語だけならその1語。§11 #8、実装時に調整）。雑談は資料と語が重ならないため、自然に「添えない」側に落ちる。
- 関連チャンクが無い場合は、雑談か業務相談かを**区別せず**「何も添えずに従来どおり応答する」で統一する。
  「資料にありません」を言い出さないこと。
- 検索が**何らかの理由で失敗した場合**（DB 障害・通信エラー等）も、エラーで応答を止めず、
  チャンクを添えずに通常どおり応答する（**縮退動作**）。原因を問わず挙動は同じ。

### FR-Q2-05 出典表示

- 検索でヒットして回答に添えた社内資料を、**控えめに出典を表示**する
  （AI が実際にどのチャンクを使ったかまでは判定できないため、「添えたチャンク＝参照資料」として扱う）。
  - 本文には混ぜず、吹き出しの末尾に「▶ 参照した資料」としてたたみ、クリックで開く形とする。
- 既存の Google 検索の出典表示の仕組み（`route.ts` の `SOURCES_SEP` によるストリーム末尾への JSON 付加）を
  **流用する**。表示の作法を Google 検索の出典と揃えること。
- 1つの回答に Google 検索の出典と社内資料の出典が**両方付く場合は、区別して表示**する
  （例：「▶ 参照した社内資料」「▶ Web の参照元」のように分ける。見せ方の詳細は設計工程で決める）。

### NFR-Q2-01 機密性

- 登録する資料は**社内一般レベルに限る**（社外秘・顧客情報・個人情報を含まない）。
- 資料は Gemini API（外部サービス）に送信されるため、**これは運用ルールとして守る必要がある**。
  登録前に内容を確認することを PR レビューの観点に含める。

### NFR-Q2-02 コスト

- キーワード検索は**埋め込み API を使わない**ため、取り込み・検索とも埋め込みの無料枠上限
  （1日1,000リクエスト）と**無関係**。埋め込みコストが発生するのはベクトル後乗せ時のみ
  （その際は取り込みの間隔調整・リトライと #95 の再開フラグを使う）。
- 検索結果を毎回プロンプトに添えるため**1回あたりの生成トークンが増える**。添えるチャンク数に上限（4件）を設ける。
- 現在の無料枠の1日上限（v1.0 で 429 対応済み）に当たりやすくなる可能性があるため、
  上限に達したときの既存メッセージが引き続き機能すること。

### NFR-Q2-03 応答開始までの時間

- キーワード検索は生成前に **DB 照会を1回挟むだけ**で、追加の遅延はごく小さい（埋め込みの往復が無い）。
- ベクトル検索を後乗せする場合は、生成前に「ベクトル化＋検索」の往復が加わり最初の応答が遅くなるため、
  **その時点で実測し、許容範囲を再評価する**。
- v1.0 NFR-01（ストリーミング・「考え中…」表示）が、生成開始後の体感と無応答時間を引き続きカバーする。

## 6. Q3：「一般論 → 内田論」の相談体験

### FR-Q3-01 2段構えの回答（一般論 → 内田さんの考え）

- 社内資料が回答の根拠になるときは、回答を**次の2つの見出し**で構成する
  （1つの回答の中で見出しを分ける。別ボタンでの深掘りにはしない）。

```
## 一般論          …世間一般ではどう考えられているか（1〜3文で短く）
## 内田さんの考え   …資料に基づく内田さんならではの見方（こちらが本題）
```

- 見出しの文言はこの2つに**固定**する（増やさない・変えない）。呼び方は「内田さんの考え（内田論）」に統一。
- 資料が根拠にならないとき（雑談・該当なし）は、この形式を使わず従来どおり応答する（FR-Q3-02）。

> 補足（2026-07-22 変更）：当初の「常に見出しで2段」を、人格プロンプトの『結論から短く・自然に会話』
> 方針と衝突して安定しなかったため、いったん「違いが伝わればよい」に緩めた（Issue #82）。
>
> 補足（2026-07-23 再変更）：レビューの結果、**見える2段構えを求める方針に戻した**。前回不安定だった
> 原因は「見出しを使わず、地の文で自然に2段」という**繊細な語りの指示**が AI に難しかったこと。今回は
> 逆に**見出し2つの固定フォーマットを明示指定**する（形式の遵守は AI が最も得意とする指示形態のため、
> 安定が見込める）。それでも安定しない場合は、区切りトークン＋画面側での分割表示（構造的保証）に切り替える。

### FR-Q3-02 該当する資料が無いとき

- 社内資料に該当が無い場合は**一般論との対比をせず、従来どおり応答する**（FR-Q2-04 と同じ扱い。
  雑談か業務相談かの区別判定は行わない）。
- 内田さんの見方（内田論）を**創作しない**。無いものを作らないこと。
- 「経験談か一般論かを区別して伝える」は v1.0 の PS-06「知ったかぶりをしない」および
  直近の「作り話をしない」方針が引き続きカバーする（本書で新たな明記義務は設けない）。

### FR-Q3-03 内田さんの見方（内田論）の忠実さ

- 「内田さんならではの見方」として出す内容は、**事実は資料に忠実**とする。
- ただし言い回し・たとえ方は内田さんらしくしてよい（**資料を軸に口調を足す程度は許容**）。
- 資料に無い体験談を創作してはならない。

### FR-Q3-04 重要な判断のハンドオフ

- 本番環境の操作・セキュリティ・お金や契約・評価や人間関係など**重い判断**は、
  資料の有無にかかわらず断定せず、内田さん本人や担当者への確認を勧める（v1.0 PS-07 を継続）。

## 7. Q4：AI駆動開発の相談窓口として機能

### FR-Q4-01 相談できる範囲

次の**両方**に答えられること。

| 種類 | 例 |
| --- | --- |
| Claude Code の使い方 | 「Claude Code でこういうときどうする？」「このエラーの意味は？」 |
| AI を使った開発の進め方 | 「要件定義って何を書くの？」「この順番でいい？」 |

- ツールの対象は **Claude Code に絞る**（研修テキストが Claude 編、`CLAUDE.md` も Claude Code 前提で、
  「うちのやり方」として答えられる知識が Claude Code のものだけのため）。
- 他の AI ツール（Copilot・Cursor 等）は対象外。聞かれた場合は資料が無いものとして一般論で答える（FR-Q4-02）。

### FR-Q4-02 知識との連携

- 社内の開発ルール（`CLAUDE.md` の開発工程・ブランチ運用・公開手順など）を**知識として登録**し、
  「うちのやり方」で答えられるようにする。
- Claude Code 自体の最新仕様は変化が速いため、資料に無いことは**従来どおり一般論として答える**
  （FR-Q3-02 と同じ扱い。「うちのやり方」をでっち上げない）。

### FR-Q4-03 実装方針

- 原則として**人格プロンプト（`uchidaPrompt.ts`）の調整と知識登録で実現**する。
- 新しい画面や機能の追加は行わない。

## 8. 制約・前提

- CN-01 pgvector 拡張は**ベクトル後乗せ時に使用**（有効化済み：`scripts/sql/knowledge.sql`。キーワード検索では未使用）。
- CN-02 Gemini の埋め込み API（`gemini-embedding-001`）も**ベクトル後乗せ時のみ使用**。キーワード検索は埋め込みを使わない。
- CN-03 取り込みコマンド（`npm run ingest`）はローカル実行を前提とする。
  実行者は Supabase への書き込み権限（`SUPABASE_SERVICE_ROLE_KEY`）が必要。
- CN-04 v1.0 の制約（Next.js 16 / React 19 / TypeScript / Tailwind 4 / Vercel / Supabase Auth）を継続する。
- CN-05 `main` への直接 push は禁止。各条件ごとに作業ブランチを切り、PR 経由でマージする（`CLAUDE.md`）。

## 9. 既知の課題（v1.0 から継続）

- KI-01 ~~`POST /api/assistant` が認証を確認していない~~ → **解消済み**（Issue #76 / PR #85 マージ済み）。
  未認証は 401 で断り、社内知識が外部に漏れない。あわせて KI-03（呼びかけ名がクライアント申告値だった問題）も同 PR で解消。
- KI-04 レート制限・利用量上限がない。
- KI-05 会話が長いほどトークンが増える。RAG のチャンク追加で影響が大きくなる。
- KI-06 人格の再現度を測る仕組みがない。

> KI-01 は「Q2（#77）公開の前提条件」として先に対応した（Issue #76）。Q2 で社内資料を扱い始めると、
> 未認証アクセスのリスクが「API コストの消費」から「社内知識の漏えい」に変わるため、先行して解消した。

## 10. 受け入れ基準（クリア判定チェックリスト）

判定は自分でこのチェックリストを埋めることで行う。会話の品質に関する項目の試験質問には、
`docs/uchida-interview-sheet.md` の想定問答（18問）を流用する。ただし Q2〜Q4 向けの質問
（資料に該当がある質問／無い質問、AIツール・開発ルールの質問）はカバーされていないため、数問追加する。

### Q1 親しみやすいキャラクター表現（#74 / PR #75 で対応済み。本番での最終確認用）
- [ ] `/assistant` にキャラクターアイコンが表示される
- [ ] 待機／考え中で表情が変わる（生成中はバッジが点滅する）
- [ ] 辛い話題の応答では 🌶 バッジに変わる
- [ ] 「本人ではありません」の注記が引き続き表示される
- [ ] 「視差効果を減らす」設定時にアニメーションが止まる

### Q2 ナレッジ化（RAG）
- [ ] `docs/knowledge/` の Markdown を取り込むコマンドが動く
- [ ] ingest を**2回連続**で実行し、1回目の後と2回目の後で `knowledge` の**行数・各行の内容が完全に一致**する（冪等）
- [ ] 資料に書かれている内容を質問すると、その内容に沿った回答が返る
- [ ] 回答の末尾に、参照した資料が控えめに表示される
- [ ] 雑談（例：「辛いもの好きなんですか？」）では資料の話を持ち出さない
- [ ] 資料を修正・削除して再実行すると、`knowledge` の内容もそれに追随する
- [ ] 登録した資料に社外秘・顧客情報・個人情報が含まれていない
- [ ] 未認証（ログインしていない状態）で `POST /api/assistant` を直接呼ぶとエラーになり、
      応答が得られない（KI-01 解消。**Q2 公開の前提条件**）

### Q3 一般論 → 内田論
- [ ] 資料が根拠になる質問で、回答が「## 一般論」「## 内田さんの考え」の**2つの見出し**に分かれて表示される
- [ ] 雑談・資料に該当がない質問では、この2見出し形式に**ならない**（従来どおりの応答）
- [ ] 資料に該当がない相談では、通常の回答が返る（「内田さんの考え」が捏造されない）
- [ ] 資料に無い体験談が創作されていない
- [ ] 重要な判断では本人・担当者への確認を勧める

### Q4 AI駆動開発の相談窓口
- [ ] Claude Code の使い方の質問に答えられる
- [ ] 開発の進め方（工程・順番）の質問に答えられる
- [ ] `CLAUDE.md` の開発ルールに沿った「うちのやり方」で答える
- [ ] 資料に無いことについて「うちのやり方」を作らず、一般論として答える

### 全体（アウトカム：相談窓口として選ばれたか）
- [ ] 内田さんと研修内容を知っている人が、Q2〜Q4 の試験質問（想定問答）の回答を読み、
      「一般論ではなく、**内田さんの見方になっている**」と判断できる（例：5問中4問以上）。
      §1 の課題「答えが一般論に留まる」を裏返した、ゴール達成の確認（KI-06 の軽量版）。

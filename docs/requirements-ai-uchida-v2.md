## 1. 背景・ねらい

現状の AI内田さんは「内田さんの口調で答える AI」だが、次の弱点がある。

- **見た目がキャラクターとして立っていない**（アイコンに表情の仕組みはあるが、小さくて存在感が弱い）
- **社内の知識を持っていない**。研修で語られた内容や内田さん本人の考えを知らないため、
  答えが「どこにでもある一般論」に留まる
- そのため「内田さんに相談している」感が薄く、**相談窓口として選ばれにくい**

クエストカードの要望は次のとおり。

> もっと人間に寄せて親しみやすく。研修や内田さんの情報を吸い上げ、
> 一般論から内田さんの論まで相談できるように

これを実現し、AI内田さんを「冷たい神託」ではなく「隣に座る師匠」にする。

## 3. 用語（今回はじめて出るもの）

| 用語 | 意味 |
| --- | --- |
| RAG | AI に社内資料を検索させ、**その内容を根拠に答えさせる仕組み**。Retrieval（検索）+ Generation（生成）の略 |
| ベクトル | 文章を数百個の数字の並びに変換したもの。**意味が近い文章どうしは数字も近くなる**ため、言葉が違っても意味で探せる |
| 埋め込み（Embedding） | 文章をベクトルに変換する処理。今回は Gemini の埋め込み API を使う |
| チャンク | 資料を検索しやすい大きさに切り分けた文章のかたまり |
| 取り込み（Ingest） | 資料を読み込み → 切り分け → ベクトル化 → DB に保存する一連の処理 |
| pgvector | PostgreSQL でベクトルを扱う拡張機能。Supabase で有効化できる |

## 5. Q2：研修・内田さん情報のナレッジ化（RAG）

今回いちばん重い条件。ここが Q3・Q4 の土台になる。

### FR-Q2-01 知識の元ネタ

| 元ネタ | 状態 |
| --- | --- |
| 研修の資料・議事録 | 数百ページ以上。**形式は Markdown / テキスト**に統一する |
| 内田さん本人へのインタビュー | `docs/uchida-interview-sheet.md` が既にある。今後追加取材も行う |

- **PDF / PowerPoint / Word は対象外**とし、登録前に Markdown へ変換しておく運用とする。
  （変換処理を作らずに済み、開発量を大きく減らせるため）
- Markdown への変換と、機密が含まれないことの確認（NFR-Q2-01）は、**資料を登録する本人が行う**。
  この手作業は今回のクエストの作業量に含めて見積もる（数百ページ規模のため軽くはない）。

### FR-Q2-02 知識の置き場所と登録方法

- 資料はリポジトリ内（例：`docs/knowledge/`）に Markdown ファイルとして置き、**PR 経由で更新**する。
- 資料を追加・変更したら、**手動でコマンドを実行**して取り込み処理を走らせる（例：`npm run ingest`）。
  - 自動化（CI での実行）は今回行わない。**何が起きているか見えやすくするため**、まず手動から始める。
- 取り込み処理は**同期型**とする：実行後の `knowledge` の中身が、そのとき `docs/knowledge/` にある資料の内容と
  **一致する**こと。具体的には次の3つを満たす。
  - 同じ資料のまま再実行しても、内容が重複しない
  - 資料を**修正**して再実行すると、古いチャンクが新しい内容に置き換わる
  - 資料を**削除**して再実行すると、そのファイル由来のチャンクも消える（古い情報を根拠に答え続けない）

### FR-Q2-03 知識の保存先

- Supabase に**新規テーブル**を作る（例：`knowledge`）。既存の `conversations` / `messages` とは
  **役割が異なるため別テーブル**とする。

| テーブル | 中身 | 持ち主 | 用途 |
| --- | --- | --- | --- |
| `conversations` / `messages`（既存） | 利用者と AI のやりとり | ユーザーごと（本人のみ閲覧） | 続きから相談する |
| `knowledge`（新規） | 資料のチャンク＋ベクトル | 全員共通 | AI が答えの根拠として検索する |

- `knowledge` はログイン済みユーザーが**読み取りのみ**可能とする（書き込みは取り込み処理のみ）。
- 詳細な列定義は設計工程で決める。最低限、元ファイル名・見出し・本文・ベクトルを保持する。

### FR-Q2-04 検索と回答生成

1. ユーザーの発言を受け取る
2. 発言をベクトル化し、`knowledge` から**意味の近いチャンクを数件取得**する
3. **関連度がしきい値以上のチャンクだけ**を人格プロンプトに添えて Gemini に渡す
4. しきい値以上のチャンクが1件も無ければ、**何も添えずに従来どおり応答する**

- 注意：ベクトル検索は雑談でも「一番近いもの」を必ず返すため、検索結果が自然に空になることはない。
  「添えない」判断は**しきい値による足切り**（未確定事項 #3）で実現する。
- 関連チャンクが無い場合は、雑談か業務相談かを**区別せず**「何も添えずに従来どおり応答する」で統一する。
  「資料にありません」を言い出さないこと。

### FR-Q2-05 出典表示

- 回答の根拠に社内資料を使った場合、**控えめに出典を表示**する。
  - 本文には混ぜず、吹き出しの末尾に「▶ 参照した資料」としてたたみ、クリックで開く形とする。
- 既存の Google 検索の出典表示の仕組み（`route.ts` の `SOURCES_SEP` によるストリーム末尾への JSON 付加）を
  **流用する**。表示の作法を Google 検索の出典と揃えること。
- 1つの回答に Google 検索の出典と社内資料の出典が**両方付く場合は、区別して表示**する
  （例：「▶ 参照した社内資料」「▶ Web の参照元」のように分ける。見せ方の詳細は設計工程で決める）。

### NFR-Q2-01 機密性

- 登録する資料は**社内一般レベルに限る**（社外秘・顧客情報・個人情報を含まない）。
- 資料は Gemini API（外部サービス）に送信されるため、**これは運用ルールとして守る必要がある**。
  登録前に内容を確認することを PR レビューの観点に含める。

### NFR-Q2-02 コスト

- 埋め込み処理は資料追加・変更時のみ走るため、継続的なコストは小さい。
  ただし**初回の一括取り込みは数十万トークン規模**（数百ページ想定）になる見込み。1回きりだが、
  埋め込み API のレート制限（生成 API とは別枠）に当たる可能性があるため、取り込み処理に間隔調整・リトライを入れる。
- 検索結果を毎回プロンプトに添えるため**1回あたりのトークンが増える**。添えるチャンク数に上限を設ける。
- 現在の無料枠の1日上限（v1.0 で 429 対応済み）に当たりやすくなる可能性があるため、
  上限に達したときの既存メッセージが引き続き機能すること。

## 6. Q3：「一般論 → 内田論」の相談体験

### FR-Q3-01 一般論と社内の見方の違いを示す

- 社内資料が根拠になり、かつ**一般的な考え方と社内／内田さんの見方が食い違うとき**は、その違いが分かるように答える。

```
一般的にはこう言われがち → でも、うちの場合は（社内資料に基づく見方）
```

- 見た目を厳密に2段に割ることは求めない。内田さんの口調のまま、地の文で自然に織り込む形でよい。
- 一般論と社内の見方がほぼ同じで対比の意味が薄いときは、無理に分けず、資料に沿って自然に答える。

> 補足（2026-07-22 変更）：当初は「常に見出しで2段に割る」を条件にしていたが、人格プロンプトの
> 『結論から短く・自然に会話』方針と衝突し、賢いモデル（gemini-2.5-flash）でも安定して2段にできなかった。
> 実際の回答は資料に忠実で品質も十分だったため、"見た目の2段"ではなく"違いが伝わること"を条件に緩めた（Issue #82）。

### FR-Q3-02 該当する資料が無いとき

- 社内資料に該当が無い場合は**2段構えにせず、従来どおり応答する**（FR-Q2-04 と同じ扱い。
  雑談か業務相談かの区別判定は行わない）。
- 2段目（内田さんの考え）は**出さない**。無いものを作らないこと。
- 「経験談か一般論かを区別して伝える」は v1.0 の PS-06「知ったかぶりをしない」および
  直近の「作り話をしない」方針が引き続きカバーする（本書で新たな明記義務は設けない）。

### FR-Q3-03 2段目の忠実さ

- 「内田さんの考え」として出す内容は、**事実は資料に忠実**とする。
- ただし言い回し・たとえ方は内田さんらしくしてよい（**資料を軸に口調を足す程度は許容**）。
- 資料に無い体験談を創作してはならない。

### FR-Q3-04 重要な判断のハンドオフ

- 本番環境の操作・セキュリティ・お金や契約・評価や人間関係など**重い判断**は、
  資料の有無にかかわらず断定せず、内田さん本人や担当者への確認を勧める（v1.0 PS-07 を継続）。

## 7. Q4：AI駆動開発の相談窓口として機能

### FR-Q4-01 相談できる範囲

次の**両方**に答えられること。

| 種類 | 例 |
| --- | --- |
| AI ツールの使い方 | 「Claude Code でこういうときどうする？」「このエラーの意味は？」 |
| AI を使った開発の進め方 | 「要件定義って何を書くの？」「この順番でいい？」 |

### FR-Q4-02 知識との連携

- 社内の開発ルール（`CLAUDE.md` の開発工程・ブランチ運用・公開手順など）を**知識として登録**し、
  「うちのやり方」で答えられるようにする。
- AI ツール自体の最新仕様は変化が速いため、資料に無いことは**従来どおり一般論として答える**
  （FR-Q3-02 と同じ扱い。「うちのやり方」をでっち上げない）。

### FR-Q4-03 実装方針

- 原則として**人格プロンプト（`uchidaPrompt.ts`）の調整と知識登録で実現**する。
- 新しい画面や機能の追加は行わない。

## 8. 制約・前提

- CN-01 Supabase で **pgvector 拡張を有効化**する必要がある（未実施なら設計工程で確認する）。
- CN-02 埋め込みには Gemini の埋め込み API を使う。既存の `GEMINI_API_KEY` を流用する。
- CN-03 取り込みコマンドはローカル実行を前提とする。実行者は Supabase への書き込み権限が必要。
- CN-04 v1.0 の制約（Next.js 16 / React 19 / TypeScript / Tailwind 4 / Vercel / Supabase Auth）を継続する。
- CN-05 `main` への直接 push は禁止。各条件ごとに作業ブランチを切り、PR 経由でマージする（`CLAUDE.md`）。

## 9. 既知の課題（v1.0 から継続。KI-01 のみ Q2 公開までに解消する）

- KI-01 `POST /api/assistant` が認証を確認していない（**優先度：高**）。RAG により社内資料が回答に含まれるため、
  **未認証で叩けると社内知識が外部に漏れる**。実装は別 Issue とするが、
  **KI-01 の解消を Q2 公開（`main` へのマージ）の前提条件とする**（Q2 の受け入れ基準にも含める）。
- KI-04 レート制限・利用量上限がない。
- KI-05 会話が長いほどトークンが増える。RAG のチャンク追加で影響が大きくなる。
- KI-06 人格の再現度を測る仕組みがない。

> KI-01 の実装自体は別 Issue だが、**Q2 で社内資料を扱い始めることでリスクの重さが変わり、
> Q2 の公開可否を左右する前提条件になる**ため、本書に明記しておく。

- [ ] 同じコマンドを2回実行しても知識が重複しない
